# 面试讲解稿

## 一句话介绍

Evidra 是一个企业知识库 RAG 评测与调优工作台。它不只支持文档问答，还支持检索策略切换、查询改写、多路召回、Rerank、Prompt 配置、评测集对比和 Langfuse/本地 trace 观测。

## 为什么不是普通 RAG Demo

普通 RAG Demo 通常只展示“上传文档、embedding、相似度检索、调用大模型回答”。Evidra 更强调工程闭环：

- 能看见每次回答用了哪些证据。
- 能切换 TF-IDF、BM25、Embedding、Hybrid。
- 能用查询改写处理口语化问题。
- 能用多路召回扩大候选证据。
- 能用 Rerank 提升证据排序。
- 能用评测集比较不同策略，而不是凭感觉调参。
- 能用 trace 和 feedback 记录线上问答质量。

## 核心链路

```text
文档上传
  -> 文本抽取
  -> 分块
  -> 可选 embedding
  -> 本地索引
  -> 用户提问
  -> 可选查询改写
  -> 单路或多路召回
  -> 可选 rerank
  -> 取 Top K 证据
  -> 回答模型生成带引用答案
  -> trace / feedback / evaluation
```

## 可以重点展示的技术点

1. **可调 RAG pipeline**
   检索模式、Top K、score threshold、Prompt version、query rewrite、multi-route、rerank 都可以从 UI 切换。

2. **多供应商兼容**
   回答模型、embedding 模型和 rerank 模型都通过兼容接口配置，不绑定单一模型厂商。

3. **安全配置**
   前端不会回显密钥，只展示“已配置”占位符；运行配置文件也不会进入交付包。

4. **可观测性降级**
   配置 Langfuse 时记录远端 trace；未配置或失败时写入本地 JSONL，保证 demo 不因为观测服务失败而中断。

5. **评测驱动**
   使用 `data/eval_sets/demo.json` 做回归测试和策略对比，能说明“为什么这次改动变好或变差”。

## 演示话术

可以这样开场：

> 我做的不是一个简单聊天框，而是一个 RAG 评测与调优助手。企业知识库场景里，真正难点不是把答案生成出来，而是知道答案为什么对、为什么错、怎么稳定优化。所以我把检索、重排、Prompt、模型配置、评测和观测都放在一个可复现的工作台里。

展示多路召回时可以说：

> 用户问题往往很口语，所以我先保留原始问题，再可选用大模型改写出更适合检索的 query。多路召回会让原始问题和改写 query 分别走关键词、BM25、Embedding 和 Hybrid，再合并去重。如果配置了 rerank，就在生成答案前对候选证据重新排序。

展示评测时可以说：

> 每次调参之后，我不靠主观感觉判断效果，而是跑固定评测集，比较通过率、引用覆盖率和延迟。这让 RAG 优化变成一个可复现的实验过程。

## 后续可扩展方向

- 引入 LLM Judge，评估 answer faithfulness。
- 根据低质量 feedback 自动沉淀回归评测集。
- 生成策略实验报告。
- 支持更多向量库或生产级存储。
- 支持团队共享 Prompt 版本和评测集版本。
